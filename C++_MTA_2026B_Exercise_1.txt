Requirements and Guidelines


The exercises in the course require you to implement an educational math game, as a console application. The game is a competition between two players on solving math exercises. Exact rules follow below.


Environment and Submission Instructions:


The exercise should be implemented in Visual Studio 2022 or later, with standard C++ libraries and run on Windows with Console screen of standard size (80*25), using gotoxy for printing at a specific location on screen (X, Y), using _kbhit and _getch for getting input from the user without blocking, and Sleep for controlling the pace of the game.
(All the above would be explained to you during one of the lab sessions).
Submission is in MAMA, as a single zip file containing only the code and the vcxproj and sln files + readme.txt with IDs -- but without the DEBUG folder and without any compiled artifact.
(Please make sure to delete hidden directories and unnecessary binary files).


Note that you MUST provide a readme.txt file that would contain the IDs of the students submitting the exercise. Even if there is only a single submitter, you still need to put your ID in the readme.txt file. Your readme file will also list any decisions that you took and want to elaborate about.

Exercise 1
In this exercise you will implement an educational math game, as a console application.


You will determine how to use the console screen size (80×25 characters) for:
   * Presenting the exercise.
   * Displaying game status (point, lives).
   * The area for collecting numbers.


When the game begins, there are two player characters, placed on screen. A math exercise then appears on top of the screen, same exercise for both players. Both players will try to solve the exercise, by moving on screen and collecting digits.
Digits and characters will appear on screen randomly, at random positions:
   * digits (0 - 9) - to allow solving the exercise
   * the character ‘e’ - to erase last digit collected by this player
   * the character ‘c’ - to clear the entire number for this player
   * the character ‘@’ - to erase last digit collected by the opponent player
   * the character ‘#’ - to clear the entire number for the opponent player
   * the character ‘$’ - add points to this player
   * the character ‘*’ - this player loses life, number for this player is cleared
   * the character ‘!’ - the opponent player loses life, opponent’s number is cleared
   * The character ‘^’ - the current player’s speed is doubled, for 40 game cycles


When a new character is added, it cannot be added adjacent to another character in any direction from the 4 adjacent directions: above, below, to the right or to the left.
Also, a new character cannot appear in a distance of less than 3 points from any of the players (diff_x + diff_y from a player should be >= 3).
A possible way to find a place for a new char: generate a random position, if it is not a valid position repeat 3 times, after which continue without placing this char.


Once the game starts, the player chooses a movement direction using the designated keys, as defined below. After a direction is selected, the player continues moving automatically in that direction without requiring additional key presses, until the STAY key is pressed (as defined below). Players’ “movement speed” is 1 unless the player picks the double speed character as mentioned above. Note that double speed doesn’t mean skipping chars!!


Input must be case-insensitive (both lowercase and uppercase letters should be accepted).

Menu
The game shall have the following entry menu:
(1) Start a new game
(8) Present instructions and keys
(9) EXIT


Going beyond the edges of the screen
There is an invisible tunnel that takes the players to the other side of the screen.
(Top ⇔ Bottom, Left ⇔ Right).


Protecting the exercise and legend areas
It is the game’s responsibility (i.e. your code) to add some kind of a “Wall” char that does not allow the players to get into the exercise and legend areas.


Players characters
Player 1: A   Player 2: B


Collecting a digit
The digit is added to the solution position of this player, from left to right.


When a player solves an exercise
A message will appear on screen, the game will wait for a few moments, and the player will get additional points. Then, if the player reached the “winning score” another message will appear, announcing the winner, and the game will go back to the main menu.
If the player didn’t reach the “winning score”, the screen will be cleared, a new exercise will appear and the players will start a new match.
You should decide how many points each solution gives, how many points the char $ adds and what is the winning score.

Pausing a game
Pressing the ESC key during a game pauses the game. You should present a message on screen saying: “Game paused, press ESC again to continue or H to go to the main menu”.
When the game is at a pause state, pressing ESC again would continue the game, with the players moving exactly as before the pausing, as if the game hadn’t paused. Pressing in pause mode ‘H’ or ‘h’ (the letter H standing for “Home”) ends the current game returning to the main menu (see comments about avoiding a recursive flow).


Notes to be aware of:
   1. Do not use the command ‘exit’:
Using the command exit for finishing the program is not allowed. In general using ‘exit’ is a bad sign. You should finish the program by normally finishing main.
   2. Make sure your program does not perform unnecessary bad recursive calls. For example, if your menu calls “game.run()” for starting a game, and then the game calls “menu.run()” for presenting the menu after the game ends, this sounds like a bad recursion that should be avoided.

Error Handling
The program should never crash. In case of problems in files, always prefer overcoming the problem and allowing using the file. However if a file cannot be used, provide a proper message to the user, clearly pointing at the problem, and return from main.
